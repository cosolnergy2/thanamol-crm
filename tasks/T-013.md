### Objective

Build CRUD API endpoints for the three core CRM entities: Customer, Contact, and Company.

Endpoints for each entity:
- GET /api/customers — list with pagination (page, limit, total), search by name/email, filter by type and status
- GET /api/customers/:id — get by ID with related contacts and company
- POST /api/customers — create with Box type validation
- PUT /api/customers/:id — update with Box type validation
- DELETE /api/customers/:id — soft delete (set status to INACTIVE) or hard delete

- GET /api/contacts — list with pagination, filter by customer_id, is_primary
- GET /api/contacts/:id — get by ID with related customer
- POST /api/contacts — create with Box type validation
- PUT /api/contacts/:id — update; enforce only one is_primary per customer
- DELETE /api/contacts/:id — delete

- GET /api/companies — list with pagination, search by name/tax_id, filter by industry and status
- GET /api/companies/:id — get by ID with related customers
- POST /api/companies — create with Box type validation
- PUT /api/companies/:id — update
- DELETE /api/companies/:id — delete

Customer-specific:
- Filter by type: INDIVIDUAL, CORPORATE
- Filter by status: ACTIVE, INACTIVE, PROSPECT
- Include contacts_count in list response

Contact-specific:
- Filter by customer_id
- is_primary flag management (only one primary per customer)

Company-specific:
- Filter by industry, status
- Search by name and tax_id

All endpoints require JWT auth middleware. Use Elysia t.Object() for all request body and query validation. Validation errors must return structured messages that are useful to the frontend (field-level errors).

Reference base44 pages for expected data shapes: CustomerList.jsx, CustomerCreate.jsx, ContactList.jsx, CompanyList.jsx at `/Users/jettapat/personal/7tapat/projects/propertyflow-crm/src/pages/`.

Acceptance criteria:
- All 15 endpoints respond correctly with proper HTTP status codes
- Pagination returns page, limit, total, data array
- Search and filter query params work independently and in combination
- Box validation rejects invalid payloads with field-level error messages
- All endpoints reject unauthenticated requests with 401
- Unit tests cover happy path and validation error cases for each endpoint

### Detail

**Implementation approach:**
- Add Customer, Contact, Company DTO types in `packages/shared/src/types/index.ts` with pagination response shapes
- Implement `apps/api/src/routes/customers.ts` with full CRUD using Prisma `customer` model; list includes `_count.contacts`; search uses `contains` on name/email/phone with `mode: insensitive`
- Implement `apps/api/src/routes/contacts.ts`; POST/PUT enforce `is_primary` uniqueness per customer via a Prisma `updateMany` to unset others before setting the new primary
- Implement `apps/api/src/routes/companies.ts`; Company model uses plain string for status (not enum) — validate with `t.Optional(t.String())`
- Register all three route plugins in `apps/api/src/index.ts`
- Write unit tests mocking Prisma, following the pattern from `roles.test.ts`
