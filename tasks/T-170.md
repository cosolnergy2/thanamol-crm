# T-170: PO detail/edit + status workflow

**Status:** In Progress
**Assignee:** dev
**Created:** 2026-04-01
**Updated:** 2026-04-01

### Objective

Build the PO detail view and edit form, and implement the Issue / Receive / Close / Cancel status workflow. Currently the PO detail page does not display the new fields added in T-169 (conditions, documents, delivery address, PO type). The status workflow actions are missing so POs cannot progress through their lifecycle.

Acceptance criteria:
- Detail page displays all PO fields: header (PO number, vendor, company, site, unit, PO type, PO date, payment due date, delivery address), item lines table with quantity / unit price / total, Conditions section as read-only summary, Documents list, linked PR reference
- Edit form mirrors the T-169 create form and pre-fills with existing PO data; only allowed when PO is in Draft status
- Issue button on Draft POs — moves status to Issued
- Receive button on Issued POs — moves status to Fully Received
- Close button on Fully Received POs — moves status to Closed
- Cancel button on Draft or Issued POs — prompts for reason, moves status to Cancelled
- Status badge on detail page updates immediately after each workflow action

### Detail

**Implementation approach:**
- Add `POST /:id/close` endpoint to the API (FULLY_RECEIVED -> CLOSED transition)
- Add `useClosePurchaseOrder` hook in `usePurchaseOrders.ts`
- Rewrite `apps/web/src/routes/_authenticated/facility-management/procurement/orders/$poId/index.tsx` to display all PO fields (General Info, Items table with all columns, Conditions read-only summary, Documents list, Close workflow button)
- Add cancel-with-reason modal using a simple dialog state (prompt user for reason text before calling cancel)
- Create edit page at `apps/web/src/routes/_authenticated/facility-management/procurement/orders/$poId/edit.tsx` mirroring the create form, pre-filled with existing PO data; guarded to DRAFT-only

**Files to create or modify:**
