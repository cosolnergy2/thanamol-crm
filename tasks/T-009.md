**Status:** Done
**Assignee:** dev

### Objective

Build the role-based access control API layer in `apps/api`. This includes full CRUD for roles, user-role assignment endpoints, a permissions aggregation endpoint, and reusable Elysia permission middleware.

**Endpoints to implement:**

| Method | Path | Description |
|--------|------|-------------|
| GET | /api/roles | List all roles |
| POST | /api/roles | Create a role (name, description, permissions JSON) |
| PUT | /api/roles/:id | Update a role |
| DELETE | /api/roles/:id | Delete a role |
| POST | /api/users/:userId/roles | Assign a role to a user |
| DELETE | /api/users/:userId/roles/:roleId | Remove a role from a user |
| GET | /api/users/:userId/permissions | Get the user's aggregated permissions (union of all role permissions) |

**Permission middleware:**

Create a reusable Elysia guard `requirePermission(permission: string)` that:
- Reads the authenticated user from the JWT context (set by existing auth middleware)
- Fetches the user's aggregated permissions
- Returns 403 with `{ error: 'Forbidden' }` if the user does not have the required permission
- Can be composed on any endpoint: `.use(requirePermission('manage_users'))`

**Permissions storage:**
- The `Role` model already has a `permissions` JSON field in the Prisma schema
- Permissions are strings, e.g. `"manage_users"`, `"view_financials"`, `"manage_properties"`

**Acceptance criteria:**
1. All 7 endpoints respond correctly with proper status codes
2. POST /api/roles validates name is required (400 if missing)
3. `requirePermission` middleware returns 403 when user lacks permission
4. `requirePermission` middleware passes through when user has permission
5. Unit tests cover all endpoints and the middleware (both allow and deny cases)

### Detail

**Implementation approach:**
- Create `apps/api/src/middleware/permissions.ts` — a reusable Elysia plugin factory `requirePermission(permission)` that fetches the user's roles from DB, aggregates all `permissions` JSON fields, and returns 403 if the required permission key is not `true`
- Create `apps/api/src/routes/roles.ts` — full CRUD for Role model using `authGuard` for all routes and `requirePermission('manage_roles')` for write operations; DELETE prevents removal if UserRole records exist
- Add user-role assignment endpoints (`POST /api/users/:userId/roles`, `DELETE /api/users/:userId/roles/:roleId`, `GET /api/users/:userId/permissions`) inside `roles.ts` under the `/api/users` prefix
- Register both route groups in `apps/api/src/index.ts`
- Write unit tests in `apps/api/src/routes/roles.test.ts` covering all 7 endpoints and both allow/deny cases of the permission middleware
