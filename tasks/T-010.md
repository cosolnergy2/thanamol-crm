### Objective

Build the frontend permission layer in `apps/web`. This task produces three exports: a data-fetching hook and two render-guard components that conditionally show UI based on what the current user is allowed to do.

**Deliverables:**

1. **`usePermissions()` hook** (`apps/web/src/hooks/usePermissions.ts`)
   - Calls `GET /api/users/:userId/permissions` via TanStack Query
   - Returns `{ permissions: string[], roles: string[], hasPermission(p: string): boolean, hasRole(r: string): boolean, isLoading, isError }`
   - Cache key: `['permissions', userId]`
   - Reads userId from `AuthContext`

2. **`PermissionGuard` component** (`apps/web/src/components/PermissionGuard.tsx`)
   - Props: `permission: string`, `children: ReactNode`, `fallback?: ReactNode`
   - Renders children only if `hasPermission(permission)` is true
   - Renders `fallback` (or nothing) otherwise

3. **`RolePermissionGuard` component** (`apps/web/src/components/RolePermissionGuard.tsx`)
   - Props: `role: string`, `children: ReactNode`, `fallback?: ReactNode`
   - Renders children only if `hasRole(role)` is true

### Detail

**Implementation approach:**
- `usePermissions` hook uses TanStack Query with cache key `['permissions', userId]` to fetch `GET /api/users/:userId/permissions`, which returns `{ permissions: Record<string, boolean> }`. Roles come from `currentUser.roles` (already in AuthContext). Returns `{ permissions, roles, hasPermission, hasRole, isLoading, isError }`.
- `PermissionGuard` wraps `usePermissions` and conditionally renders `children` or `fallback` based on `hasPermission(permission)`.
- `RolePermissionGuard` wraps `usePermissions` and conditionally renders based on `hasRole(role)`. Both guards render `null` (not a loading spinner) when loading to keep them lightweight and composable.
- `TopBar` gets a user avatar/name dropdown with Profile link and Logout button using `useAuth()`.
- `Sidebar` already filters nav items via `userHasPermission`; the settings nav item is gated by `permission: 'settings'` — this is kept as-is since the existing role-based filter handles it.
