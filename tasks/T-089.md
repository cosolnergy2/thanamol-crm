# T-089: Add Officer/Other to inventory category dropdown

**Status:** In Progress
**Assignee:** dev
**Created:** 2026-04-07
**Updated:** 2026-04-07

### Objective

The inventory category dropdown is purely database-driven. It is missing "Officer" and "Other" options that already exist on the Asset form via the `EXTRA_ASSET_CATEGORIES` constant pattern. Users cannot assign these categories when creating or editing inventory items.

Fix by adding an `EXTRA_INVENTORY_CATEGORIES = ['Officer', 'Other']` constant and applying it in the inventory create and edit forms, following the same pattern used in the asset forms.

Acceptance criteria:
- `EXTRA_INVENTORY_CATEGORIES` constant is defined (in `packages/shared` or co-located, matching the asset pattern)
- Inventory create form category dropdown shows "Officer" and "Other" options in addition to database-driven options
- Inventory edit form category dropdown shows the same options and correctly pre-selects the saved value
- `pnpm build` passes with zero errors
- `pnpm test` passes

### Detail

Implementation approach:
- Add `EXTRA_INVENTORY_CATEGORIES = ['Officer', 'Other'] as const` constant to both inventory forms, co-located after `ITEM_STATUS_OPTIONS`, matching the pattern in `assets/create.tsx` line 45.
- In each form's category `<SelectContent>`, append the extra options after the database-driven category items.
- No backend or DB changes needed — the values are stored as plain strings and the `__none__` guard in `handleSubmit` already handles non-UUID category values correctly.

Files modified:
- `apps/web/src/routes/_authenticated/facility-management/inventory/create.tsx`
- `apps/web/src/routes/_authenticated/facility-management/inventory/$itemId/edit.tsx`
