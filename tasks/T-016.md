### Objective

Build CRUD API endpoints for the three property management entities: Project, ProjectTemplate, and Unit.

Project endpoints:
- GET /api/projects — list with pagination, search by name, filter by status and type
- GET /api/projects/:id — get by ID with summary stats (unit counts by status)
- POST /api/projects — create with Box type validation
- PUT /api/projects/:id — update
- DELETE /api/projects/:id — soft delete
- GET /api/projects/:id/units — list units belonging to a project (with filter/pagination)
- GET /api/projects/:id/dashboard — stats: total_units, available, reserved, sold, booked; revenue summary

ProjectTemplate endpoints:
- GET /api/project-templates — list all templates
- GET /api/project-templates/:id — get by ID
- POST /api/project-templates — create
- PUT /api/project-templates/:id — update
- DELETE /api/project-templates/:id — delete

Unit endpoints:
- GET /api/units — list with pagination, filter by project_id, status, type, floor
- GET /api/units/:id — get by ID with project and current booking/contract info
- POST /api/units — create with Box type validation
- PUT /api/units/:id — update
- DELETE /api/units/:id — soft delete
- GET /api/units/availability — availability matrix: grouped by project and floor, shows status per unit

All endpoints require JWT auth. Use Elysia t.Object() for request validation with field-level error messages.

Reference base44 pages for expected data shapes: ProjectList.jsx, ProjectDetail.jsx, UnitList.jsx, UnitAvailability.jsx at `/Users/jettapat/personal/7tapat/projects/propertyflow-crm/src/pages/`.

Acceptance criteria:
- All CRUD endpoints respond with correct HTTP status codes
- Project dashboard endpoint returns unit counts by status and revenue totals
- Unit availability endpoint returns a matrix suitable for grid rendering (grouped by project/floor)
- Pagination returns page, limit, total, data array
- Unit tests cover happy path and validation errors for each endpoint group

### Detail

**Implementation approach:**
- Project routes use `authPlugin` + `guard` pattern (matches `roles.ts`). Soft delete sets `status = SUSPENDED` (no deleted_at column in schema).
- Unit availability endpoint `/api/units/availability` must be registered BEFORE `/api/units/:id` to avoid the param route capturing "availability" as an id.
- Dashboard stats computed with `prisma.unit.groupBy` on `status` field, revenue total summed with `prisma.unit.aggregate`.
- Shared types added to `packages/shared/src/types/index.ts` — Project, Unit, ProjectTemplate DTOs with pagination envelope.
- All tests follow the `roles.test.ts` pattern: vi.mock prisma, sign JWT via helper, assert HTTP status + response shape.
