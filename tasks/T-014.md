Technical requirements:
- Use TanStack Router file-based routes under `apps/web/src/routes/`
- Use TanStack Query for all data fetching and mutations (with loading/error states)
- Use UI components from T-006 (Table, Card, Button, Input, Select, Badge, Dialog, etc.)
- Use React Hook Form for CustomerCreate form — include field-level validation messages
- Loading states must use skeleton components, not spinners alone
- Error states must show actionable messages (not generic "Something went wrong")
- Responsive design: tables must be horizontally scrollable on mobile

Acceptance criteria:
- All 4 pages render data from the API (T-013)
- Tables support sorting by column, pagination with page size selector
- Search input debounces before firing API request
- Filters update the query and reset to page 1
- CustomerCreate form validates required fields, shows inline error messages, and redirects to list on success
- Badge colors match status/type values using design tokens (indigo/teal palette)
- No hardcoded colors or arbitrary Tailwind values

### Detail

**Implementation approach:**
- Create shared data hooks (useCustomers, useContacts, useCompanies) that wrap TanStack Query with typed API calls
- Build reusable components: PageHeader, StatusBadge — composed from existing ui/ primitives
- Port 4 route files under `_authenticated/` using TanStack Router `createFileRoute`; replace React Router patterns and base44 client with our apiGet/apiPost/apiPut/apiDelete
- CustomerList uses server-side pagination+search+filter via query params; ContactList and CompanyList use client-side filtering to stay consistent with base44 behaviour given smaller data sets
- CustomerCreate uses React Hook Form + zod validation for field-level inline errors; success triggers sonner toast + navigate to /customers
