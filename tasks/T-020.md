# T-020: API — Lead, Deal CRUD + pipeline stages

**Status:** In Progress
**Assignee:** dev
**Created:** 2026-03-15
**Updated:** 2026-03-15

### Objective
Build CRUD endpoints for Lead and Deal entities with pipeline stage management.

Lead endpoints:
- GET /api/leads — list with pagination, search, filter by status/source/assignedTo
- GET /api/leads/:id — with customer, contact, assigned user relations
- POST /api/leads — create with validation
- PUT /api/leads/:id — update (including stage transitions)
- DELETE /api/leads/:id

Deal endpoints:
- GET /api/deals — list with pagination, filter by stage/assignedTo
- GET /api/deals/:id — with customer, lead relations
- GET /api/deals/pipeline — grouped by stage with counts and total values
- POST /api/deals — create
- PUT /api/deals/:id — update (stage transitions)
- DELETE /api/deals/:id

Acceptance criteria:
- All endpoints return proper HTTP status codes and error messages
- Pagination works on list endpoints (page, limit query params)
- Pipeline endpoint returns stages with deal counts and total deal values
- Stage transitions are validated (only allowed transitions accepted)
- All validation uses Elysia Box type system (t.Object()), not Zod
- Shared types for Lead and Deal entities live in packages/shared/src/types/

Reference: LeadList.jsx, LeadDetail.jsx, DealPipeline.jsx, DealDetail.jsx

### Detail

**Implementation approach:**
- Follow the same Elysia route pattern established in `customers.ts`: `authPlugin` + `guard` for auth, `t.Object()` for request validation
- Lead status transitions allowed: NEW -> CONTACTED -> QUALIFIED/UNQUALIFIED -> CONVERTED; define as a constants map
- Deal stage transitions allowed: PROSPECTING -> QUALIFICATION -> PROPOSAL -> NEGOTIATION -> CLOSED_WON/CLOSED_LOST; define as a constants map
- Pipeline endpoint groups deals by stage using a Prisma `groupBy` query with `_sum` on value and `_count`
- Shared types (Lead, Deal, LeadStatus, DealStage, etc.) added to `packages/shared/src/types/index.ts`

**Files created/modified:**
- `apps/api/src/routes/leads.ts` — Lead CRUD routes
- `apps/api/src/routes/deals.ts` — Deal CRUD routes + pipeline endpoint
- `apps/api/src/routes/leads.test.ts` — unit tests for leads
- `apps/api/src/routes/deals.test.ts` — unit tests for deals
- `apps/api/src/index.ts` — register leadsRoutes and dealsRoutes
- `packages/shared/src/types/index.ts` — Lead, Deal shared types

**API endpoints:**
- GET /api/leads — paginated list; query: page, limit, search (title), status, source, assignedTo
- GET /api/leads/:id — lead with customer, contact, assignee
- POST /api/leads — create lead; body: title (required), customerId, contactId, source, status, value, probability, expectedCloseDate, notes, assignedTo
- PUT /api/leads/:id — update lead (includes status transitions)
- DELETE /api/leads/:id
- GET /api/deals — paginated list; query: page, limit, stage, assignedTo
- GET /api/deals/pipeline — returns `{ stage, deals, count, totalValue }[]` grouped by stage
- GET /api/deals/:id — deal with customer, lead, assignee
- POST /api/deals — create deal; body: title (required), customerId, leadId, stage, value, probability, expectedCloseDate, notes, assignedTo
- PUT /api/deals/:id — update deal (includes stage transitions)
- DELETE /api/deals/:id
