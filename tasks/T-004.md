### Objective

Build JWT-based authentication API to replace Base44 auth. This is the backend auth layer that all protected API routes will depend on.

**Endpoints required:**

| Method | Path | Description |
|--------|------|-------------|
| POST | /api/auth/register | Register new user (email, password, first_name, last_name) |
| POST | /api/auth/login | Login with email/password, return access_token + refresh_token |
| POST | /api/auth/refresh | Refresh access token using refresh_token |
| GET | /api/auth/me | Get current user profile (requires auth) |
| POST | /api/auth/logout | Invalidate refresh token |

**Technical requirements:**
- Use `@elysiajs/jwt` plugin for token signing and verification
- Use `bcryptjs` for password hashing and comparison
- Access token expires in 15 minutes
- Refresh token expires in 7 days
- Store refresh tokens in database (add `RefreshToken` model to Prisma schema if not present)
- Create `authGuard` middleware that verifies JWT from `Authorization: Bearer <token>` header and attaches user to context

**Reference:** `/Users/jettapat/personal/7tapat/projects/propertyflow-crm/src/lib/AuthContext.jsx` — for understanding the expected auth flow from the client side.

**Acceptance criteria:**
1. All 5 endpoints respond with correct status codes and payloads
2. JWT tokens are correctly signed and verified using the configured secret
3. `authGuard` middleware returns 401 for missing or invalid tokens
4. Invalid credentials on login return 401 with a clear error message
5. Refresh token rotation works (old token invalidated on refresh)
6. Unit tests cover happy path and error cases for all endpoints
